Java环境介绍

Java语言、JVM和生态系统

Java编程环境是由Java语言和Java虚拟机(JVM)组成的。微软的.NET环境也沿用了非常类似的平台架构，然而.NET比Java晚了几年才出现。耐人寻味，不过Java是站在巨人的肩膀上诞生的，.NET也是如此罢了(¬‿¬)

Java成功的一个主要原因是，整个生态系统是个标准的环境。这意味着组成Java环境的各种技术都有规范，不同的技术提供方之间的组件能够实现相互兼容。

Java原由Sun公司开发，现已被甲骨文公司(Oracle)收购。红帽、IBM、惠普、SAP、苹果和富士通等公司也大量参与了 Java 标准技术的实现。

OpenJDK是Java的开源版本，由多家公司合作开发。

Java 由多个不同但相互联系的环境和规范组成，包括 Java 移动版(Java ME)、Java 标准版(Java SE)和 Java 企业版(Java EE又称J2EE)。

JVM是什么

JVM是一个程序，它提供了运行Java程序所需的运行时环境。换言之，如果某个硬件和操作系统平台没有相应的JVM，就无法运行Java程序。

提供给JVM的并不是Java语言源码，而是经编译成的Java字节码(.class)文件。JVM是字节码格式程序的解释器，一次只执行字节码中的一个指令，JVM从.class的字节码文件中读出一条指令，翻译一条指令，然后执行一条指令，这个过程就称为Java的解释执行。

所以，可以说Java既是编译型语言，也是解释型语言，因为它同时具备编译性和解释性。

上述也能解释为什么Java是跨平台的编程语言，“一次编写，到处运行”，意思就是Java类文件可以从一个运行平台迁移到另一个平台，只要有可用的JVM就能正常运行。

JVM的设计方式建立在几个早期编程环境的多年发展经验之上，尤其是C/C++，因此有多个设计目的，都是为了减轻程序员的负担。

  • 包含一个容器，让应用代码在其中运行
  • 较之C/C++，提供了一个更安全的执行环境
  • 代开发者管理内存
  • 提供了一个跨平台的执行环境

除了上述主要设计目的之外，还有一个比较特殊的点，那就是JVM能够使用运行时的信息进行自我管理。就是说，JVM会收集运行时信息，从而对如何执行代码做出更好的决定，即监控并优化运行中的程序。一个典型的例子，在运行Java程序的生命周期中，各组成部分被调用的次数并不都是相同的，有些部分调用的次数远比其他部分多得多，Java平台使用JIT(just-in-time compilation)的技术解决这个问题。在HotSpot JVM(Sun为Java 1.3 开发的JVM)中，JVM首先识别程序的哪一部分调用最频繁(这一部分称为“热点方法”)，然后跳过JVM解释器，直接把热点方法编译成机器码。

JVM利用可用的运行时信息，让程序的性能比纯粹经过解释器执行更高。

描述JVM必须怎样运行的标准叫做JVM规范

Java程序的生命周期

整个流程从Java源码开始，经过javac程序处理后得到类文件，这个文件中保存的是编译源码后得到的Java字节码。类文件是Java平台能处理的最小功能单位，也是把新代码传给运行中程序的唯一方式。

新的类文件通过类加载机制载入到虚拟机，从而把新类型提供给解释器执行。

一些概念

  1. 字节码是什么？

    字节码是一种处于源码和机器码之间的“中间表现形式”。字节码的目的是，提供一种能让JVM解释器高效执行的格式

  2. javac是编译器吗？

    编译器一般是生成机器码，而javac生成的是和机器码不太一样的字节码。不过，类文件有点像对象文件（例如Windows中的.dll文件，或Unix中的.so文件），人类读不懂。javac更像是编译器的“前半部分”，它生成的中间表现形式可以进一步处理，生成机器码。只有JIT编译才会生成机器码

  3. Java是解释性语言吗？

    JVM基本上算是解释器，但是大多数的解释型语言都是直接从源码解释程序，而JVM解释器需要的是类文件。Java源码是通过编译转换成.class字节码文件的，而JVM对字节码文件是解释执行的，所以说Java同时具备编译性和解释性

  4. 其他语言可以在JVM中运行吗？

    可以。JVM可以运行任何有效的类文件，Java之外的语言可通过两种方式在JVM中运行。第一种，提供用于生成类文件的源码编译器（类似于javac），以类似Java代码的方式在JVM中运行（Scala等语言采用的就是这种方式）。第二，还可以使用Java实现解释器和运行时，然后解释该语言使用的源码格式（JRuby等语言采用的就是这种方式）。
